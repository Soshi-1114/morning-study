### Q1. `[HTTP/Web]`（初級）

- [ ] 答えられた

**CORS（Cross-Origin Resource Sharing）** とは何か。「なぜ必要か」という背景（Same-Origin Policy）と、サーバー側の基本的な対応を説明してください。

<details>
<summary>解答を見る</summary>

**Same-Origin Policyとは：** ブラウザのセキュリティ機能で、あるオリジン（プロトコル＋ドメイン＋ポートの組み合わせ）のJavaScriptが、別オリジンのリソースに自由にアクセスできないよう制限する仕組み。悪意のあるサイトが勝手に別サービスのAPIを叩いてユーザーデータを盗むのを防ぐ。

**CORSとは：** この制限を安全に「緩める」仕組み。サーバーがレスポンスヘッダで「このオリジンからのアクセスは許可する」と宣言することで、ブラウザが通信を許可する。

**基本的な対応（サーバー側）：**
```
Access-Control-Allow-Origin: https://example.com  # 許可するオリジンを指定
Access-Control-Allow-Methods: GET, POST, PUT       # 許可するメソッド
Access-Control-Allow-Headers: Content-Type         # 許可するヘッダ
```

**なぜ大事か：** フロントエンド（例：`app.example.com`）とAPIサーバー（例：`api.example.com`）が別ドメインの場合、CORS設定がないとブラウザがリクエストをブロックする。`*`（全許可）は手軽だが、認証付きリクエストには使えず、本番では許可オリジンを絞るべき。

</details>

### Q2. `[データベース]`（初級）

- [ ] 答えられた

**N+1問題** とは何か。ユーザーとその投稿を取得する例を使って説明し、解決策も述べてください。

<details>
<summary>解答を見る</summary>

**N+1問題とは：** 1件の一覧取得クエリ（1回）の後、各レコードに対して個別のクエリをN回発行してしまうパフォーマンス問題。計N+1回のクエリが発行される。

**具体例：**
```
# 悪い例（N+1）
users = User.all          # クエリ1回：全ユーザー取得
users.each do |user|
  user.posts              # クエリN回：ユーザーごとに投稿取得
end
# ユーザーが100人いれば101回のクエリが発生
```

**解決策：JOINやEager Loadingでまとめて取得する**
```
# 良い例（Eager Loading）
users = User.includes(:posts).all  # クエリ2回（usersとpostsをまとめて取得）
```

**なぜ深刻か：** ユーザー100人なら101クエリ、1000人なら1001クエリと線形に増える。各クエリにDB接続コストがかかるため、本番でリスト画面が急に重くなる原因の定番。SQLのスローログやO/Rマッパーのクエリログで検出できる。

</details>

### Q3. `[セキュリティ]`（初級）

- [ ] 答えられた

**CSRF（クロスサイトリクエストフォージェリ）** とはどのような攻撃か。攻撃の仕組みと基本的な対策を説明してください。

<details>
<summary>解答を見る</summary>

**CSRFとは：** 認証済みのユーザーを騙して、意図しないリクエストを正規サービスに送らせる攻撃。

**攻撃の仕組み：**
1. ユーザーが銀行サイト（`bank.com`）にログイン中（セッションCookieが有効）
2. 攻撃者が罠サイトに `<img src="https://bank.com/transfer?to=attacker&amount=100000">` を仕込む
3. ユーザーが罠サイトを訪問 → ブラウザが自動でCookieを付けてbank.comへリクエストを送信
4. 銀行サーバーは「正規ユーザーからのリクエスト」と誤認し送金を実行

**基本的な対策：**
1. **CSRFトークン**：フォームにサーバーが発行したランダムなトークンを埋め込み、送信時にサーバーが照合する（攻撃者はトークンを知ることができない）
2. **SameSite Cookie属性**：`SameSite=Strict` or `Lax` を設定し、クロスオリジンリクエストにCookieを送らないようにする（現在最も強力な対策）
3. **Refererチェック**：リクエスト元のオリジンを確認（補助的な対策）

**XSSとの違い：** XSSはスクリプトを「実行させる」攻撃、CSRFは認証済みの「リクエストを送らせる」攻撃。

</details>

### Q4. `[設計/アーキテクチャ]`（初級）

- [ ] 答えられた

**依存性の注入（Dependency Injection / DI）** とは何か。注入しない場合との比較を交えて説明し、メリットを述べてください。

<details>
<summary>解答を見る</summary>

**DIとは：** クラスが使う依存オブジェクト（他のクラスやサービス）を、クラス内で`new`するのではなく、外から「渡してもらう」設計パターン。

**DI未使用（悪い例）：**
```python
class OrderService:
    def __init__(self):
        self.email = GmailEmailService()  # 具体的な実装に依存
    def create_order(self, order):
        self.email.send(...)
```
→ メール業者をSendGridに変えたい → OrderServiceを直接修正しなければならない

**DI使用（良い例）：**
```python
class OrderService:
    def __init__(self, email_service):  # 外から渡してもらう
        self.email = email_service
    def create_order(self, order):
        self.email.send(...)

# 使う側で注入
service = OrderService(email_service=SendGridEmailService())
```

**メリット：**
- **テストしやすい**：本物のメールサービスをモックに差し替えてテストできる
- **変更に強い**：依存先の実装を変えてもOrderService本体を触らなくてよい
- **疎結合**：クラス同士の結びつきが弱まり、それぞれを独立して変更・再利用できる

</details>

### Q5. `[プログラミング基礎]`（初級）

- [ ] 答えられた

**ハッシュテーブル**（連想配列・ディクショナリ）とは何か。なぜ検索がO(1)になるのかを、ハッシュ関数とセットで説明してください。

<details>
<summary>解答を見る</summary>

**ハッシュテーブルとは：** キーと値のペアを格納するデータ構造。キーをハッシュ関数に通して「格納場所（インデックス）」を決め、そこに直接アクセスする。

**O(1)になる理由：**
```
key → ハッシュ関数 → インデックス番号 → 配列の該当位置に直接アクセス
"apple" → hash("apple") → 42 → array[42] を参照
```
全要素を順番に探さず、キーから「場所」を計算して直接飛べるので、データ数に関係なく原則1回のアクセスで済む。

**ハッシュ衝突とは：** 異なるキーが同じインデックスになること。対処法はチェーン法（同じインデックスにリストで連結）やオープンアドレッシング（空き場所を探す）など。衝突が多いと最悪O(n)になる。

**実用例：**
- プログラミング言語の辞書型（Python の `dict`、JavaScript の `Object/Map`、Ruby の `Hash`）
- キャッシュ（URLをキーにして結果を保存）
- データベースのハッシュインデックス（等値検索に特化）

</details>
